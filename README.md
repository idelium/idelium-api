# Idelium API

Idelium API is the Laravel backend for the Idelium test automation platform. It
stores projects, environments, reusable steps, test cases, test cycles, platform
catalogues, and execution results. It serves both the browser application and
[`idelium-cli`](https://github.com/idelium/idelium-cli).

This repository contains the API only. Use
[`idelium-web`](https://github.com/idelium/idelium-web) for the administration
interface and [`idelium-docker`](https://github.com/idelium/idelium-docker) to
run the complete local stack.

## Main capabilities

- Cookie-authenticated administration API for the Idelium Web application.
- API-key-authenticated execution API for Idelium CLI agents.
- Multi-customer projects, accounts, environments, plugins, tests, and cycles.
- Selenium test imports and remote test launch requests.
- Browser, operating-system, device, model, brand, and location catalogues.
- Storage and retrieval of performed cycles, tests, steps, and Postman results.
- Database-backed sessions and queue processing in the container deployment.

## Architecture

```text
Idelium Web ── Sanctum session + CSRF ──┐
                                        ├── Idelium API ── MariaDB
Idelium CLI ───── Idelium-Key header ───┘          └────── queue worker
```

The API follows the standard Laravel request flow: routes invoke controllers,
Form Requests validate input, policies and ownership checks authorize access,
models persist data, and API resources or explicit response objects serialize
the result. Tenant ownership must be enforced for every customer-owned lookup
and mutation; client-supplied identifiers are never authorization evidence.

## Requirements

- PHP 8.2, 8.3, or 8.4.
- Composer 2.10.2.
- Required PHP extensions for Laravel and the selected database driver.
- MariaDB/MySQL for normal development and production, or SQLite for tests.

Dependency resolution uses a PHP 8.2 Composer platform. `composer.lock` is
committed so local, CI, and container installations resolve the same packages.

## Local setup

Clone the repository, install the locked dependencies, and create local
configuration:

```bash
git clone https://github.com/idelium/idelium-api.git
cd idelium-api
composer install
cp .env.example .env
php artisan key:generate
```

Create a database and update the `DB_*` values in `.env`, then initialize it:

```bash
php artisan migrate
```

Start the development server and, when asynchronous jobs are being exercised, a
queue worker in another terminal:

```bash
php artisan serve
php artisan queue:work
```

The default application URL is `http://localhost:8000`. The frontend origin
must be explicitly allowed through `CORS_ALLOWED_ORIGINS`.

For an integrated HTTPS environment with the database and frontend already
wired together, follow the
[`idelium-docker` quick start](https://github.com/idelium/idelium-docker#quick-start-demo).

## Configuration

Start from `.env.example`. Never commit `.env`, application keys, database
passwords, API keys, access tokens, session identifiers, or authorization
headers.

### Application

| Variable | Purpose | Local example |
| --- | --- | --- |
| `APP_ENV` | Laravel environment | `local` |
| `APP_KEY` | Laravel encryption key generated by Artisan | generated value |
| `APP_DEBUG` | Detailed error pages; disable outside development | `true` |
| `APP_URL` | Canonical API origin | `http://localhost:8000` |

### Database and runtime services

Configure `DB_CONNECTION`, `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME`,
and `DB_PASSWORD` for MariaDB/MySQL. Cache, session, and queue drivers are chosen
with `CACHE_STORE`, `SESSION_DRIVER`, and `QUEUE_CONNECTION`. The Docker stack
uses database-backed sessions and runs the queue worker as a managed process.

### Browser sessions and CORS

Browser authentication uses Laravel Sanctum's stateful session mode:

- `CORS_ALLOWED_ORIGINS` is a comma-separated allow-list of trusted frontend
  origins. Credentials are supported, so wildcard origins must not be used.
- `SANCTUM_STATEFUL_DOMAINS` can override the hosts treated as first-party.
- `SESSION_SECURE_COOKIE=true` requires HTTPS and is expected in deployment.
- `SESSION_SAME_SITE=lax` is the default cross-site request policy.
- `SESSION_LIFETIME` controls the session lifetime in minutes.

See [Browser session authentication](docs/api/browser-session-authentication.md)
for the complete login, CSRF, logout, and compatibility contract.

### Optional integrations

Mail, Redis, AWS storage, reCAPTCHA, and broadcast configuration use Laravel's
standard environment variables and the corresponding files under `config/`.
Leave optional integrations disabled or empty when they are not used. Any
credential must be injected by the deployment secret provider, never embedded
in source or Compose configuration.

## Authentication

### Browser application

The supported sequence is:

1. Request `GET /api/sanctum/csrf-cookie` with credentials enabled.
2. Submit `POST /api/login` with the CSRF cookie and header.
3. Call protected `/api/*` routes using the resulting secure, HttpOnly session
   cookie.
4. End the session with `POST /api/logout`.

The API deliberately does not expose a session identifier or bearer token to
browser JavaScript. OAuth2/Passport endpoints are not part of this service.

### CLI agents

Routes under `/api/ideliumcl` require the customer API key in the `Idelium-Key`
request header. The middleware resolves the owning customer and downstream
queries must remain scoped to that customer. Keep this key in a protected secret
store and never put it in source, command output, URLs, or support logs.

## API areas

The route definitions in `routes/api.php` are the authoritative endpoint list.
The main areas are:

| Area | Authentication | Responsibility |
| --- | --- | --- |
| Session | CSRF/session | Login, logout, and current-user data |
| Navigation | Session | Sidebar, header, and selected-customer context |
| Administration | Session | Roles, accounts, customers, profiles, and API-key rotation |
| Test design | Session | Projects, cycles, tests, ordered steps, environments, and plugins |
| Execution | Session | Test launch, Selenium import, and performed-result inspection |
| Platforms | Session | Status, type, OS, browser, device, model, brand, and location data |
| CLI configuration | API key | Cycle, test, step, plugin, and environment retrieval |
| CLI reporting | API key | Performed cycle, test, and step creation or update |

Response and ownership contracts are documented separately:

## Database changes

Create schema changes with Laravel migrations. Migrations must be deterministic,
safe for existing data, and reversible whenever practical. API, configuration,
database, or persisted-data changes must remain backward compatible or include
an explicit migration and release note.

Useful development commands are:

```bash
php artisan migrate:status
php artisan migrate
php artisan migrate:rollback --step=1
php artisan migrate:fresh --seed
```

`migrate:fresh` destroys the selected database and is only appropriate for an
isolated development or test database.

## Tests and quality gates

Behavioral changes require tests at the lowest useful level. Authorization and
tenant-isolation changes must include negative tests proving that a resource
owned by another customer cannot be read or modified.

Run the same gates used by CI from the repository root:

```bash
composer install --no-interaction --prefer-dist
composer validate --strict --no-check-publish
composer audit --locked --no-interaction
find app bootstrap config database routes tests -name '*.php' -print0 | xargs -0 -n1 php -l
composer format:check
composer analyse
touch database/ci.sqlite
APP_ENV=testing APP_KEY=base64:AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA= DB_CONNECTION=sqlite DB_DATABASE=database/ci.sqlite php artisan migrate:fresh --force
APP_ENV=testing APP_KEY=base64:AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA= DB_CONNECTION=sqlite DB_DATABASE=database/ci.sqlite php artisan migrate:rollback --step=1 --force
APP_ENV=testing APP_KEY=base64:AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA= DB_CONNECTION=sqlite DB_DATABASE=database/ci.sqlite php artisan migrate --force
APP_ENV=testing APP_KEY=base64:AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA= DB_CONNECTION=sqlite DB_DATABASE=:memory: composer test
```

With PCOV or Xdebug enabled, enforce the statement-coverage gate:

```bash
APP_ENV=testing APP_KEY=base64:AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA= DB_CONNECTION=sqlite DB_DATABASE=:memory: vendor/bin/phpunit --coverage-clover=coverage.xml
composer coverage:check
```

CI runs the main test suite on PHP 8.2, 8.3, and 8.4 and enforces at least 60%
statement coverage in its dedicated coverage job.

To apply formatting locally, run `vendor/bin/pint`. Review the resulting diff;
generated files and unrelated source must not be changed.

## Container deployment

The supported full-stack deployment is maintained in
[`idelium-docker`](https://github.com/idelium/idelium-docker). It builds this
repository at an exact Git revision, installs locked production dependencies,
injects secrets through files, runs migrations in a one-shot initialization
service, and exposes the API through the frontend HTTPS reverse proxy.

For a manual deployment:

- set `APP_ENV=production`, `APP_DEBUG=false`, and a stable `APP_URL`;
- inject a unique application key and database credentials securely;
- terminate HTTPS with a trusted certificate;
- run `php artisan migrate --force` as a controlled release step;
- run the queue worker under a process supervisor;
- cache configuration and routes only after final environment values are set;
- verify health, authentication, tenant isolation, and rollback procedures.

## Security expectations

- Scope every customer-owned query to the authenticated customer.
- Centralize authentication and authorization in middleware, policies, and
  reusable query scopes.
- Validate external input with Form Requests and return standard HTTP errors.
- Use transactions for multi-record mutations.
- Return explicit response fields; never serialize secrets or internal session
  data accidentally.
- Redact credentials and sensitive request values from diagnostics.
- Keep production debug output disabled.
- Review dependency audit findings and locked dependency updates before merge.

Report suspected vulnerabilities privately to the maintainers. Do not open a
public issue containing credentials, customer data, or exploit details.

## Troubleshooting

### CSRF token mismatch or unauthenticated browser requests

Confirm that the frontend first requests the CSRF cookie, both applications use
compatible HTTPS and domain settings, Axios sends credentials, and the frontend
origin appears exactly in `CORS_ALLOWED_ORIGINS`. Clear stale browser cookies
after changing session domains.

### Database connection failures

Check the selected driver, host, port, database, and user permissions. Inside
containers, `DB_HOST` must be the Compose service name rather than `localhost`.
Do not include passwords in copied error output.

### Configuration changes do not take effect

Clear Laravel's cached state in development:

```bash
php artisan optimize:clear
```

Rebuild production caches only after the final environment and secret files are
available.

### CLI receives `401 Invalid key`

Verify that the request uses the `Idelium-Key` header, the key belongs to the
expected customer, and the key has not been rotated. Never paste the key into an
issue or diagnostic transcript.
